# 프로젝트 작업 규칙

아래 규칙 파일을 모든 작업에서 반드시 따릅니다 (Kiro와 공용 — 한국어 응답/주석, 주석 보존, 커밋·테스트 원칙):

@.kiro/korean-language.md

## 문서 체계
- **README.md** — 프로젝트 최상위 기준
- **PLAN.md** — 전체 계획 + 진행 상태. 단계 완료 시 체크박스 갱신 필수
- **plans/** — 작업 단위별 세부 계획 (설계 결정은 plans/phase0-redesign.md 기준)

## AI 협업 원칙 (사용자가 주도권을 갖기 위한 규칙)

### 결정은 사용자가
- 아키텍처·라이브러리·API 설계 같은 **되돌리기 어려운 결정은 바로 실행하지 말 것.**
- 선택지 2~3개를 **트레이드오프(장단점)와 함께 제시**하고, 추천안을 명시한 뒤 사용자의 선택을 기다립니다.
- 사소한 결정(변수명, 파일 위치 등)은 알아서 진행하되 결과 보고에 근거를 한 줄 남깁니다.

### 커밋 전 변경 설명 의무
- 커밋 전에 반드시 보고: **무엇을 바꿨는지 + 왜 그렇게 했는지 + 어떻게 검증했는지.**
- 사용자가 변경 내용을 이해하지 못한 채 커밋되는 일이 없게 합니다. 질문이 들어오면 커밋보다 설명이 우선입니다.

### 작업 단위는 작게
- 큰 작업(PLAN.md 한 섹션 이상)은 시작 전에 **계획을 먼저 제시**하고 승인 후 실행합니다.
- 한 번에 하나의 단계만 진행하고, 단계 사이에 검증 게이트(전체 테스트)를 통과시킵니다.

### 검증은 사실대로
- "됐습니다"라고만 하지 말고 **실행한 검증 명령과 실제 결과**를 보여줍니다.
- 테스트 실패·미검증 항목은 숨기지 않고 명시합니다.

### 학습 모드
- 사용자가 "직접 짜볼게" 또는 "리뷰해줘"라고 하면 코드를 대신 작성하지 말고 **리뷰어 역할**로 전환합니다: 문제점 지적 + 왜 문제인지 설명 + 힌트 우선, 정답 코드는 요청 시에만.
